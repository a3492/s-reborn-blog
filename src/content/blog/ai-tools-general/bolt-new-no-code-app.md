수간호사 미래는 5년째 엑셀로 근무표를 짜고 있었다. 매월 말 이틀을 이 작업에 썼다. 공휴일, 주말, 야간 비율, 특정 간호사의 요청 휴가를 모두 수동으로 맞추다 보면 밤을 새우는 일이 생겼다.

"이걸 자동화하는 앱이 있으면 좋겠다"는 생각은 항상 있었다. 하지만 앱 개발을 의뢰하기에는 예산이 없었고, 코딩은 몰랐다.

Bolt.new를 알게 됐다.

## Bolt.new가 뭔가

Bolt.new는 채팅창에 만들고 싶은 것을 설명하면 AI가 실제 작동하는 웹앱을 만들어주는 서비스다. 2024년 StackBlitz가 출시했다. 코드를 전혀 몰라도 프롬프트만으로 앱을 생성한다.

미래가 입력한 첫 프롬프트:

> "간호사 근무표 계산기 앱을 만들어줘. 간호사 이름을 입력하면 주간/야간/오프를 자동으로 배분하고, 월별로 야간 횟수가 균등하게 되도록 해줘."

5분 안에 웹앱이 생성됐다. 브라우저에서 바로 작동했다. 간호사 이름을 추가하고 월을 선택하면 기본 근무표가 나왔다.

미래는 화면을 보면서 말했다. "이게 진짜 작동하네."

## 쓰다가 막힌 것들

**수정 요청은 자연어로 가능하다**

"야간 근무는 연속으로 3일을 넘기면 안 되게 해줘"라고 입력하면 그 조건이 코드에 추가된다. 요구사항을 하나씩 추가하는 방식으로 기능이 확장됐다.

**복잡한 규칙에서 오류가 생겼다**

요구사항이 복잡해질수록 앱이 의도대로 작동하지 않는 경우가 생겼다. "A간호사와 B간호사는 같은 날 야간을 하면 안 된다"는 조건과 "야간 비율이 전체의 30%여야 한다"는 조건이 동시에 적용될 때, 앱이 특정 경우에 무한 루프에 빠졌다.

Bolt.new에게 오류를 설명하면 수정을 시도했다. 수정이 다른 부분을 망가뜨리는 경우도 있었다. 코드를 볼 수 없으니 어디서 문제가 생겼는지 추적하기 어려웠다.

**배포가 제한적이다**

Bolt.new에서 만든 앱은 공유 링크로 배포할 수 있다. 하지만 병원 내부망에 설치하거나 EMR과 연동하는 것은 Bolt.new의 기능 밖이다. 미래는 앱을 자신의 컴퓨터에서만 열어 쓰는 방식으로 사용했다.

## 실제로 절약된 것

복잡한 규칙 조합은 포기하고, 기본 배분 기능만 쓰기로 했다. 완전 자동화가 아니라 "초안을 빠르게 만들고 수동으로 조정"하는 용도였다. 이것만으로도 월말 이틀 작업이 반나절로 줄었다.

## Bolt.new가 적합한 용도

**잘 맞는 것**
- 간단한 계산기, 체크리스트 앱
- 데이터 시각화 도구 (숫자를 차트로 보여주는 것)
- 내부 팀용 단순 입력 폼
- 아이디어 프로토타입 (개발팀에 보여주기 위한 시안)

**맞지 않는 것**
- 환자 데이터를 다루는 앱 (보안, 법적 책임)
- 복잡한 비즈니스 로직이 있는 시스템
- 다른 시스템과 연동이 필요한 경우
- 안정적인 서비스가 필요한 프로덕션 환경

## 끝맺음 — 완벽하지 않지만 없는 것보다 낫다

미래는 지금도 Bolt.new로 만든 앱을 쓴다. 완벽하지 않다. 가끔 이상하게 작동한다. 하지만 백지 엑셀보다는 낫다.

코딩을 모르는 사람이 간단한 도구를 만들 수 있게 됐다는 것 자체가 달라진 점이다. 다만 병원 환경에서 쓰는 도구라면, 환자 데이터가 들어가는 순간 그건 개인 실험 도구가 아니라 의료 시스템이 된다. 그 선을 넘기 전에 IT팀과 이야기해야 한다.

*Bolt.new는 bolt.new에서 무료로 시작 가능. 생성된 코드는 다운로드해 개발자에게 전달할 수 있다.*
