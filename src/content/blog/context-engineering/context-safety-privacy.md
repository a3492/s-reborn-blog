## 한줄 요약
환자 데이터를 AI 컨텍스트에 넣으려면 비식별화 → 전송 암호화 → 접근 통제 → 로그 관리 → 보관 기한 설정 다섯 단계를 반드시 거쳐야 하며, 법적 기준을 충족하지 못하면 사용 자체를 중단해야 한다.

## 본문

### 의료 AI의 딜레마: 정보가 많을수록 답변이 좋지만

컨텍스트 엔지니어링의 핵심 원칙은 관련 정보를 충분히 제공할수록 AI 답변 품질이 올라간다는 것이다. 환자의 검사 결과, 과거 병력, 복용 약물을 상세히 넣으면 AI는 훨씬 구체적이고 정확한 답변을 생성한다.

그런데 이 정보는 대부분 민감한 개인정보다. 환자 이름, 주민등록번호, 진료 기록은 법으로 보호되는 정보이며, 이를 외부 AI 서비스에 전송하는 것은 법적 위반이 될 수 있다.

### 관련 법규 개요

#### 한국: 개인정보보호법 + 의료법

한국에서 의료 데이터를 AI에 전송할 때 적용되는 주요 법규는 다음과 같다.

- 개인정보보호법: 개인정보의 수집, 이용, 제3자 제공에 정보주체의 동의 필요
- 의료법 제21조: 환자의 진료 기록은 정당한 사유 없이 제3자에게 제공 금지
- 생명윤리법: 인체유래물 및 유전체 정보의 연구 목적 사용 시 IRB 승인 필요
- 데이터 3법 (2020 개정): 가명정보의 통계, 연구 목적 활용 허용 (단, 재식별 금지)

#### 미국: HIPAA

HIPAA(Health Insurance Portability and Accountability Act)는 PHI(Protected Health Information)의 보호를 규정한다.

- 18가지 식별자를 제거하면 비식별화(de-identification) 완료로 간주
- 비식별화된 데이터는 HIPAA 적용 대상이 아님
- BAA(Business Associate Agreement) 없이 PHI를 제3자에게 전송하면 위반

### 비식별화(De-identification) 기법

비식별화는 데이터에서 개인을 식별할 수 있는 정보를 제거하는 과정이다.

1단계: 직접 식별자 제거
- 이름, 주민등록번호, 전화번호, 주소
- 이메일, 의료기록번호, 보험번호

2단계: 준식별자 처리
- 생년월일 → 연령대로 변환 (예: 1958년생 → 60대 후반)
- 거주지 → 광역 단위로 일반화 (예: 서울시 강남구 → 서울시)
- 진료일 → 기간으로 변환 (예: 2025-03-15 → 2025년 1분기)

3단계: 맥락적 식별 가능성 검토
- 희귀 질환 + 특정 연령대 + 특정 지역의 조합이 개인을 특정할 수 있는지 확인
- k-익명성: 같은 조합에 해당하는 사람이 최소 k명 이상이어야 함

```
[비식별화 전]
환자명: 김OO, 주민번호: 580315-1XXXXXX
서울시 강남구 역삼동 거주
2025-03-15 서울대병원 내분비내과 초진
HbA1c 9.2%, 제2형 당뇨 진단

[비식별화 후]
ID: PT-20250315-001
60대 후반 남성, 서울시 거주
2025년 1분기 3차 병원 내분비내과 초진
HbA1c 9.2%, 제2형 당뇨 진단
```

#### 비식별화의 한계

비식별화는 완벽하지 않다.

- 의료 데이터는 조합에 의한 재식별 위험이 높다 (희귀 질환 + 특정 시술 등)
- AI 컨텍스트에 넣을 때 비식별화를 하면 임상적으로 중요한 정보가 손실될 수 있다
- 비식별화된 데이터라도 원본 데이터와 결합하면 재식별이 가능하다

### 클라우드 AI vs 온프레미스 AI

#### 클라우드 AI (GPT-4, Claude 등 API)

- 데이터가 외부 서버로 전송됨
- 제공업체의 보안 정책에 의존
- HIPAA 준수 플랜이 있는 서비스도 있지만 BAA 체결이 필요
- 한국법 기준: 개인정보의 국외 이전에 해당할 수 있음

장점: 최신 모델, 높은 성능, 별도 인프라 불필요
단점: 데이터 유출 위험, 법적 불확실성, 제3자 의존

#### 온프레미스 AI (로컬 LLM)

- 병원 내부 서버에서 모델을 직접 운영
- 데이터가 외부로 나가지 않음
- 접근 통제와 로그 관리를 직접 수행

장점: 데이터 유출 원천 차단, 법적 리스크 최소화
단점: 모델 성능 제한, 인프라 비용, 운영 인력 필요

#### 현실적 선택

대부분의 의료기관에서 현실적인 선택은 다음과 같다.

- 비식별화된 데이터로 클라우드 AI 사용: 교육, 연구, 일반적 가이드라인 질문
- 식별 데이터가 필요한 경우 온프레미스: 개별 환자 의사결정 지원

### 실전: 환자 데이터를 AI에 넣을 때 반드시 지켜야 할 5가지

1. 반드시 비식별화한 후 전송한다

클라우드 AI를 사용할 때 환자 이름, 주민번호, 연락처 등 직접 식별자는 절대 포함하지 않는다. 자동 비식별화 도구를 파이프라인에 내장하는 것이 안전하다.

2. 전송 구간은 암호화한다

API 호출 시 TLS 1.2 이상의 암호화 프로토콜을 사용한다. HTTP가 아닌 HTTPS만 허용한다.

3. 접근 권한을 최소화한다

AI 시스템에 접근할 수 있는 사람을 최소한으로 제한한다. 역할 기반 접근 제어(RBAC)를 적용하고, 의사와 관리자의 접근 수준을 구분한다.

4. 모든 조회와 질의를 로그로 남긴다

누가 언제 어떤 환자 데이터를 AI에 전송했는지 감사 로그(audit log)를 기록한다. 이 로그는 개인정보보호법상 요구되는 접근 기록이기도 하다.

5. 보관 기한을 설정하고 만료 시 삭제한다

AI 컨텍스트에 사용된 데이터는 목적 달성 후 즉시 삭제한다. API 제공업체가 데이터를 보관하지 않는다는 정책을 확인한다. OpenAI의 경우 API 사용 시 데이터를 학습에 사용하지 않는다고 명시하고 있지만, 이를 계약서(DPA)로 확인해야 한다.

### 동의 획득의 현실

법적으로 환자 데이터를 AI에 사용하려면 환자의 동의가 필요하다. 하지만 현실에서 진료 중에 "AI에 데이터를 넣어도 될까요?"라고 묻기 어렵다.

현실적인 방안은 다음과 같다.

- 진료 동의서에 AI 활용 조항을 포함 (사전 포괄 동의)
- 비식별화 후 사용 시에는 가명정보 처리로 동의 면제 가능 (데이터 3법 기준)
- 연구 목적 사용 시 IRB 심의를 통해 동의 면제 가능

## 핵심 정리

1. 한국법과 HIPAA 모두 환자 식별 정보를 외부 AI에 전송하는 것을 엄격히 규제한다
2. 비식별화는 직접 식별자 제거 + 준식별자 일반화 + 재식별 가능성 검토 3단계로 수행한다
3. 클라우드 AI는 비식별 데이터에, 온프레미스 AI는 식별 데이터에 사용하는 것이 현실적이다
4. 비식별화, 암호화, 접근 통제, 로그 관리, 보관 기한 5가지는 반드시 지켜야 한다
5. 진료 동의서에 AI 활용 조항을 사전 포함하는 것이 동의 획득의 현실적 방안이다
