## 한줄 요약
AI 답변은 첫 번에 완성될 필요가 없다 — "이 부분을 더 구체적으로", "형식을 바꿔줘" 같은 피드백을 반복하면 처음 프롬프트를 아무리 잘 써도 얻기 어려운 결과물이 나온다.

## 본문

### 완벽한 첫 프롬프트는 환상이다

많은 사람이 '완벽한 프롬프트'를 한 번에 작성하려고 한다. 그러나 실제 프롬프트 엔지니어링은:

1. 대충 요청 → 2. 결과 확인 → 3. 피드백 → 4. 수정 요청 → 5. 반복

이 사이클이 실제 작업 방식이다. 이를 반복 개선 프롬프팅(Iterative Prompting)이라고 한다.

---

### 반복 개선 사이클

1단계: 초안 요청
일단 시작한다. 완벽하지 않아도 된다.
```
당뇨 환자 교육 자료를 만들어줘.
```

2단계: 결과 평가
- 형식은 맞는가?
- 깊이가 맞는가?
- 빠진 내용은 없는가?
- 어조가 맞는가?

3단계: 구체적 피드백
```
좋아. 두 가지만 수정해줘:
1. 복약 지도 부분을 더 구체적으로 — 인슐린 보관 방법 포함
2. 전체 어조를 더 쉽게 — 중학생도 이해할 수 있는 수준으로
```

4단계: 재생성 → 반복

---

### 효과적인 피드백 문장 패턴

| 목적 | 피드백 문장 |
|------|-------------|
| 더 자세히 | "~~ 부분을 더 구체적으로 설명해줘" |
| 더 간결히 | "전체를 절반 분량으로 줄여줘" |
| 형식 변경 | "bullet 대신 표로 바꿔줘" |
| 어조 변경 | "더 딱딱하게 / 더 친근하게" |
| 관점 추가 | "환자 입장도 포함해서 다시 써줘" |
| 섹션 추가 | "마지막에 주의사항 섹션 추가해줘" |
| 틀린 내용 수정 | "3번 항목에서 용량이 틀렸어. [정확한 용량]으로 수정해줘" |

---

### 의료 실전 예시: 환자 설명문 다듬기

```
[턴 1 — 초안 요청]
고혈압에 대한 환자 교육 자료를 작성해줘.

[턴 2 — 구조 피드백]
좋은데, 섹션을 다시 나눠줘:
1. 고혈압이란? (2~3문장)
2. 왜 치료해야 하나? (합병증 중심)
3. 생활습관 교정 (bullet 5개)
4. 약 복용 안내
5. 언제 병원에 와야 하나?

[턴 3 — 어조 피드백]
전반적으로 어투가 딱딱해. "~하십시오" 대신 "~하시면 좋아요" 수준으로
부드럽게 바꿔줘. 약 복용 안내는 현재 유지.

[턴 4 — 마무리]
이제 이걸 A4 1장에 들어가는 분량으로 조정해줘.
글자 크기 고려해서 너무 촘촘하지 않게.
```

처음부터 이 모든 조건을 한 번에 넣으면 프롬프트가 복잡해지고 AI가 일부를 무시하기 쉽다. 단계별로 나누는 게 오히려 더 정확하다.

---

### "다시 해줘" vs 구체적 피드백

```
❌ "다시 해줘."
→ AI는 비슷하게 재생성한다. 무엇이 문제인지 모르기 때문.

✅ "2번 항목의 약물 용량이 틀렸어. Amlodipine은 5mg으로 시작하고 최대 10mg야. 수정해줘."
→ 정확한 부분만 고쳐서 재생성한다.
```

피드백은 무엇이, 어떻게 바뀌어야 하는지를 명시해야 한다.

---

### 빠른 반복을 위한 팁

부분 수정 요청
전체를 다시 쓰지 않아도 된다.
```
"3번 항목만 수정해줘. 나머지는 그대로 유지."
```

버전 비교 요청
```
"현재 버전(전문가용)과 환자용 버전을 나란히 보여줘."
```

반영 확인 요청
```
"방금 내가 요청한 변경사항이 다 반영됐는지 확인해줘."
```

---

### 반복 개선 프롬프팅의 한계

- 대화가 길어지면 초반 요청을 AI가 잊거나 무시하는 경우가 있다
- 여러 번 수정할수록 원본 구조가 무너질 수 있다 — 5번 이상 수정하면 처음부터 다시 쓰는 게 나을 수 있다
- 핵심 요건은 매 턴마다 재확인하는 것이 좋다

## 핵심 정리
- 첫 프롬프트가 완벽할 필요 없다 — 반복 개선이 실제 작업 방식
- 피드백은 "무엇을 어떻게" 구체적으로
- 전체를 다시 쓰지 않고 특정 섹션만 수정 요청하면 효율적
