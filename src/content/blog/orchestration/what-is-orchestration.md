> 시리즈 «RoundPrep»
> 오케스트레이션 글만 모아 두면 정의가 잘 정리된 백과사전이 된다. 읽다 지루해지는 건 그래서다. 이 카테고리 글들은 동일한 가상 프로젝트 한 줄기로 엮었다. 내과 병동에서 회진 전 브리핑을 줄이는 내부 PoC RoundPrep — 레지던트 김서진, 임상정보 이혜준. 아래 본문의 “DB에서 통계 뽑아 PDF” 예시는 그들이 상부에 처음 던졌다가 “그게 AI 한 번이면 되나?”라는 질문을 받은 요구와 이어진다.



## 들어가며

ChatGPT에 "병원 DB에서 지난달 외래 통계 뽑아 PDF로"라고만 하면 막힌다. 모델이 나쁜 게 아니라 텍스트 입출력만으로는 그 일이 정의되지 않았기 때문이다. 이 글은 여러 모델·도구·(필요하면) 에이전트를 묶어 한 목표를 끝내는 설계를 오케스트레이션이라 부를 때 무엇을 의미하는지 정리한다.

다루지 않는 것: 특정 벤더 제품 튜토리얼, 병원 정보시스템 구춡 세부 계약. 다루는 것: 의사·임상 IT가 아키텍처 대화를 할 때 쓰는 공통 어휘와 체크포인트.

읽은 뒤에는 "이 요구는 단일 프롬프트인가, 파이프라인으로 끝나는가, 아니면 오케스트레이션 층이 필요한가"를 한 번에 나눠 말할 수 있으면 충분하다.

## 한줄 요약

오케스트레이션은 여러 AI·도구·에이전트가 각자의 역할을 하면서 하나의 목표를 달성하도록 단계·의존성·상태·오류 복구를 지휘하는 기술(및 운영)이다.

## 본문

### 단일 AI의 한계를 처음 실감하는 순간

처음 ChatGPT나 Claude를 써본 의사라면 이런 경험이 있을 것이다.

"우리 병원 지난달 외래 환자 통계를 분석해서 PDF 보고서로 만들어줘."

AI는 할 수 없다고 한다. 세 가지 이유다:

1. 병원 DB에 접근하는 도구가 없다
2. 실시간 데이터를 가져올 수 없다
3. PDF를 직접 생성할 수 없다

이건 AI가 무능해서가 아니다. 단일 언어모델 호출의 구조적 한계다. 텍스트를 받아 텍스트를 내놓는 계약을 넘는 행동은 외부 도구와 다단계 흐름이 필요하고, 그 흐름을 정의·실행·감시하는 층이 오케스트레이션이다.

---

### 레지던트에게 "다 해줘"라고 말하면 어떻게 될까

신입 레지던트에게 "이 환자 전부 정리해줘"라고 하면 막막해한다. 무엇부터 해야 할지, 어디서 어떤 정보를 가져와야 할지, 결과를 어떻게 정리해야 할지.

경험 있는 시니어 레지던트는 다르다. 이 작업을 분해한다:

- EMR에서 최근 기록 조회
- Lab 수치 확인 및 추세 분석
- 현재 투약 목록 검토
- SOAP 노트 형식으로 정리

그리고 병렬로 처리할 수 있는 것(Lab 조회와 투약 목록 조회는 동시에 가능)과 순차로 해야 하는 것(분석은 데이터 수집 후에 가능)을 구분한다.

오케스트레이션은 AI 시스템에 이 시니어의 머릿속 체크리스트를 코드·그래프·정책으로 옮기는 일에 가깝다. 질문에 한 번 답하는 것이 아니라, 목표를 분해하고 도구를 고르고 순서와 병렬을 관리해 결과를 완성한다.

---

### 오케스트레이션의 4가지 핵심 요소

태스크 분해 (Task Decomposition)  
복잡한 목표를 더 작은 단계로 나눈다. "환자 안전 개선 보고서 작성"은 한 번의 생성으로 끝나지 않는다. 데이터 수집, 분석, 벤치마크 비교, 권고안 작성, 보고서 구성으로 나눠야 한다.

의존성 관리 (Dependency Management)  
어떤 단계가 다른 단계의 결과를 필요로 하는지 파악한다. 데이터 수집 전에 분석을 시작할 수 없다. 반면 여러 데이터 소스 수집은 서로 의존성이 없으면 동시에 할 수 있다.

상태 관리 (State Management)  
여러 단계를 거치는 동안 중간 결과와 컨텍스트를 유지한다. 5단계 작업 중 3단계에서 실패하면, 처음부터 다시 하는 것이 아니라 3단계부터 재시도·분기할 수 있어야 한다.

오류 복구 (Error Recovery)  
에이전트나 도구가 실패했을 때 재시도, 우회, 부분 결과 반환, 사람에게 넘기기(Human-in-the-Loop) 중 무엇을 택할지 정책으로 박아 둔다.

---

### 단순 파이프라인과 무엇이 다른가

단순 파이프라인은 흐름이 고정이다.

```
A → B → C → D
```

A가 실패하면 전체 실패. 분기 없음. 조건 없음.

오케스트레이션은 유연하다.

```
        ┌→ B ─┐
A ──────┤     ├──→ D
        └→ C ─┘
        (B, C 병렬 실행)
        A 실패 시 → E로 우회
```

동적 분기, 병렬 실행, 실패 처리, 조건부 흐름. 실제 업무가 이렇게 생겼기 때문에 오케스트레이션 층이 필요하다.

---

### 의료 현장에서 오케스트레이션이 특히 필요한 영역

임상 의사결정 지원  
"이 환자 치료 방향 제안해줘"에 단일 호출로 답하면 학습 코퍼스 기반 일반론에 머무르기 쉽다. 오케스트레이션은 이 환자의 실제 데이터(EMR, Lab, 투약)를 허용된 경로로 수집하고, 가이드라인·약물 상호작용 확인 등을 단계로 묶어 통합 권고를 만든다.

의무 기록 자동화  
구술 → 환자 데이터 대조 → SOAP 구성 → 의사 검토 → EMR 저장처럼 여러 시스템과 사람이 끼는 흐름이다.

임상 연구 지원  
코호트 추출, 문헌 검색, 예비 분석이 서로 다른 시스템에 있을 때, 중간 산출물을 유지하며 이어 붙이는 설계가 필요하다.

---

### 근거 한 줄

멀티스텝·도구 사용·에이전트를 그래프나 상태머신으로 모델링하려는 흐름은 2023~2024년경 커뮤니티와 프레임워크 문서에서 공통 주제로 정리됐다. 예를 들어 LangGraph는 순환·분기·상태를 명시적으로 다루는 이유를 제품 문서 수준에서 설명한다([LangGraph 개요](https://langchain-ai.github.io/langgraph/) — 버전·URL은 발행 이후 바뀔 수 있음). "왜 지금 오케스트레이션인가"는 단일 채팅으로는 연결·승인·재시도·감사 로그 요구를 만족시키기 어렵다는 현장 문제에서 출발한다고 보는 편이 정확하다.

---

### 반례 — 이 패턴이 틀리는 경우

한 단계 요약·한 번의 검색으로 끝나는 요청에는 오케스트레이션이 과하다. 예: "이 영상 설명을 환자용 쉬운 말로만 바꿔줘." 여기에 멀티 에이전트·그래프를 얹으면 지연·비용·디버깅만 늘고 품질 이득이 거의 없다. 복잡성 비용이 사용자 가치보다 클 때는 단일 에이전트 또는 고정 파이프라인을 유지한다.

### 실무 체크리스트

- [ ] 이 목표를 2문장 이내로 쪼갠 하위 단계 목록을 적었는가?
- [ ] 각 단계의 입력·출력 데이터 형식을 정했는가?
- [ ] 의존성(순차 vs 병렬)을 표나 그림으로 적었는가?
- [ ] 단계 실패 시 재시도 횟수·우회·중단 중 무엇을 택할지 정했는가?
- [ ] 임상·개인정보가 섞이면 사람 승인 지점(HITL) 을 어디에 둘지 정했는가?
- [ ] 운영 중 로그·추적 ID로 어느 단계에서 깨졌는지 찾을 수 있는가?
