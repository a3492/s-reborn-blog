## 월요일 아침, 같은 소회의실

지난주 금요일에 적어 둔 문장이 아직 남아 있다. **「회진 전 브리핑, 짧게」.** 김서진은 커피를 한 모금 마시고 이혜준을 봤다. 주말 동안 혜준은 슬랙에 링크만 세 개 던졌다. “에이전트 프레임워크”, “멀티에이전트 논문”, “LangGraph 튜토리얼”. 서진이 웃었다. “다 에이전트잖아.”

오전 스탠드업 겸 설계 회의가 시작되자마자 **박과장**이 물었다. “그래서 **에이전트 한 마리** 달면 끝이에요, 아니면 **오케스트레이션**이 따로 있어요?” 서진은 혀가 말랐다. 같은 단어가 세 가지로 들린다. 챗봇도 에이전트라고 하고, 툴 열 개 붙인 것도 에이전트라고 하고, 팀 협업 그림 나오면 그것도 에이전트다.

혜준이 칠판에 세 줄을 그었다. **툴 → 에이전트 → 오케스트레이터.** “한 마리”는 보통 레벨 2만 말하는 경우가 많다. “여러 마리가 돌아가게”는 레벨 3에서 말한다. 박과장이 고개를 끄덕였다. “그럼 RoundPrep은 지금 몇 레벨이에요?” 서진이 대답했다. “…아직 **한 마리로 시연**하려다가, **승인·데이터** 붙이면 3이 필요해질 것 같아요.” 방 안 공기가 조금 바뀐다. 이제 회의가 **용어 전쟁**이 아니라 **범위 협상**이 될 수 있다.

### 테제

**“에이전트”만 쓰면 회의가 끝나지 않는다. 툴·에이전트·오케스트레이터를 세 층으로 나눠 말하는 순간, 무엇을 먼저 만들지 합의할 수 있다.**

### 스테이크

레벨을 착각하면 **예산과 일정이 터진다.** 에이전트 세 마리를 데려왔는데 실제로는 오케스트레이터(실패 처리·로그·권한 분리)가 없어서, 운영 단계에서 전부 갈아엎는 경우다. RoundPrep은 아직 PoC지만, 서진은 **“한 마리면 된다”는 약속**을 실수로 하고 싶지 않다.

---

## 한줄 요약

에이전트는 자율적으로 행동하는 단일 AI(툴 선택 포함)이고, 오케스트레이션은 여러 에이전트·단계의 협력·순서·실패를 설계하는 상위 구조다.

## 본문

### 용어 혼란의 이유

"AI 에이전트"라는 단어가 너무 광범위하게 쓰인다. 검색 하나 하는 간단한 봇도 에이전트, 수십 개의 도구를 써서 복잡한 업무를 처리하는 시스템도 에이전트라고 불린다. 오케스트레이션도 때로 "멀티 에이전트 시스템"이라고 불린다.

개념을 층위별로 정리하면 훨씬 명확해진다.

### 근거 한 줄

멀티 에이전트 대화와 역할 분담을 시스템으로 묶은 초기 제안 중 하나로, Microsoft 연구진의 AutoGen 프레임워크 논문(Qingyun Wu et al., *AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation*, arXiv:2308.08155, 2023)이 여러 LLM 기반 주체가 메시지로 협업하는 구성을 학술적으로 정식화한 예다. "에이전트"가 곧 "오케스트레이션"은 아니며, 대화 규칙·종료 조건·인간 개입을 누가 짜느냐가 오케스트레이션 층이다.

---

### 3층 구조로 이해하기

```
레벨 3: 오케스트레이션 레이어
        "여러 에이전트를 언제, 어떤 순서로, 어떻게 협력시킬지"
              ↕
레벨 2: 에이전트 레이어
        "주어진 목표를 달성하기 위해 스스로 판단하고 도구를 사용"
              ↕
레벨 1: 툴/모델 레이어
        "검색, 계산, DB 조회, API 호출 같은 구체적 행동"
```

**레벨 1 - 툴(Tool)**  
특정 기능을 실행하는 함수다. 자율성이 없다. 호출되면 실행하고 결과를 반환한다.

```python
def search_drug_interaction(drug1: str, drug2: str) -> dict:
    return drug_db.query(drug1, drug2)
```

**레벨 2 - 에이전트(Agent)**  
목표를 받으면 어떤 툴을 어떤 순서로 쓸지 스스로 결정한다. 계획하고, 실행하고, 결과를 보고, 다음 행동을 조정한다.

```
목표: "이 환자에게 안전한 통증 치료제 찾아줘"

에이전트의 판단:
→ 먼저 현재 복용 약물 조회
→ 신기능 확인 (NSAIDs 사용 가능 여부)
→ 약물 상호작용 DB 검색
→ 결과 바탕으로 권고안 생성
```

**레벨 3 - 오케스트레이터(Orchestrator)**  
여러 에이전트가 협력하도록 조율한다. 어떤 에이전트에게 무슨 일을 맡길지, 결과를 어떻게 합칠지, 실패하면 어떻게 할지 결정한다.

**RoundPrep에 대입하면:** “야간 데이터 긁기 + 이상 탐지 + 요약”을 **한 에이전트**에 몰아넣을지, “수집 에이전트 / 요약 에이전트”로 나눌지는 레벨 3에서 정한다. 교수님 승인 게이트는 레벨 3 정책에 가깝다.

---

### 전환점 — “에이전트만 사면 된다”는 미끼

벤더 미팅에서 흔하다. “에이전트 플랫폼 하나면 다 됩니다.” 그런데 RoundPrep처럼 **EMR 권한·감사 로그·응급 분기**가 있으면, 플랫폼이 준비한 “에이전트”만으로는 부족하고 **누가 어떤 순서로 무엇을 호출하는지**를 병원이 짜야 한다. 그게 오케스트레이션이다.

---

### 단일 에이전트 vs 멀티 에이전트

**단일 에이전트로 충분한 경우**

- 목표가 하나의 맥락 안에서 해결됨  
- 필요한 툴이 5개 이하  
- 순차적 작업, 병렬 처리 불필요  
- 전문성이 한 도메인에 집중  

예: "이 환자 SOAP 노트 초안 작성" → 단일 에이전트로 충분

**멀티 에이전트 + 오케스트레이션이 필요한 경우**

- 서로 다른 전문 지식이 필요  
- 병렬로 처리하면 빨라지는 태스크가 있음  
- 태스크가 너무 길어서 단일 컨텍스트에 안 들어감  
- 체크 앤 밸런스가 필요 (한 에이전트가 다른 에이전트를 검토)  

예: "우리 병원 이번 분기 전체 환자 안전 보고서 작성" → 멀티 에이전트 필요

---

### 오케스트레이션 패턴 3가지

**중앙 집중형 (Centralized)**  
하나의 오케스트레이터가 모든 에이전트를 직접 지휘한다. 단순하고 제어가 쉽다. 병목이 될 수 있다.

```
오케스트레이터
    ├→ 에이전트 A
    ├→ 에이전트 B
    └→ 에이전트 C
```

**분산형 (Decentralized)**  
에이전트들이 서로 직접 대화하고 결과를 전달한다. 유연하지만 제어가 어렵다.

```
에이전트 A ↔ 에이전트 B
    ↕              ↕
에이전트 C ↔ 에이전트 D
```

**계층형 (Hierarchical)**  
오케스트레이터가 서브 오케스트레이터들을 지휘하고, 각 서브 오케스트레이터가 에이전트들을 지휘한다. 대규모 시스템에 적합.

```
메인 오케스트레이터
    ├→ 진단 서브오케스트레이터
    │       ├→ 영상의학 에이전트
    │       └→ 병리 에이전트
    └→ 치료 서브오케스트레이터
            ├→ 약물 에이전트
            └→ 수술 에이전트
```

---

### 반례 — 이 구조가 틀리는 경우

오케스트레이션은 강력하지만 복잡성도 함께 온다. 무조건 멀티 에이전트를 쓰는 게 정답이 아니다.

- 에이전트 간 조율 비용이 실제 작업보다 클 때  
- 디버깅이 너무 어려워질 때  
- 실패 지점이 너무 많아질 때  
- 팀이 시스템을 이해하고 유지보수할 수 있어야 할 때  

"할 수 있다"와 "해야 한다"는 다르다.
