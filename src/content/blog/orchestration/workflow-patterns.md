## 한줄 요약
AI 오케스트레이션에는 체인, 병렬, 라우팅, 오케스트레이터-서브에이전트, 평가-최적화 패턴이 있다 — 문제에 맞는 패턴을 고르는 것이 설계의 핵심이다.

## 본문

### 패턴 1: 체인 (Chain)

가장 단순한 패턴. AI 출력이 다음 AI의 입력이 된다.

```
[AI 1] → [AI 2] → [AI 3] → 최종 출력
```

**의료 예시**: 환자 증상 → AI1(감별진단) → AI2(치료 지침 검색) → AI3(보고서 작성)

**언제 쓰나**
- 각 단계가 이전 단계의 출력만 필요할 때
- 처리 순서가 명확하고 고정일 때
- 작업이 단순하고 선형일 때

**주의점**: 체인이 길어질수록 초기 오류가 증폭된다. 1단계에서 틀리면 끝까지 틀린다.

---

### 패턴 2: 병렬화 (Parallelization)

독립적인 태스크를 동시에 실행하고 결과를 합친다.

```
            ┌→ [AI A] ─┐
[입력] ─────┤           ├→ [취합] → 출력
            └→ [AI B] ─┘
```

두 가지 변형:
- **섹션 병렬화**: 큰 문서를 나눠서 동시 처리 후 합침
- **투표(Voting)**: 같은 질문을 여러 AI에게 물어보고 다수결

**의료 예시 (투표)**:
```
같은 X-ray를 3개의 AI에게 독립적으로 판독
→ AI1: "폐렴 의심"
→ AI2: "폐렴 의심"
→ AI3: "정상"
→ 2:1로 폐렴 의심 승리
```

**언제 쓰나**
- 태스크 간 의존성이 없을 때
- 속도가 중요할 때
- 신뢰성을 높이려고 여러 관점이 필요할 때

---

### 패턴 3: 라우팅 (Routing)

입력 분류에 따라 다른 전문 처리 경로로 보낸다.

```
[입력] → [분류기] → {
    내과 케이스 → [내과 AI]
    응급 케이스 → [응급 AI]
    행정 문의 → [행정 AI]
}
```

**의료 예시**: 환자 문의가 들어오면 먼저 "이게 의학적 질문인가, 예약 문의인가, 처방전 문의인가" 분류 후 적절한 시스템으로 라우팅.

**언제 쓰나**
- 다양한 입력 유형이 존재할 때
- 각 유형에 최적화된 처리가 필요할 때
- 전문화가 일반화보다 중요할 때

---

### 패턴 4: 오케스트레이터-서브에이전트 (Orchestrator-Subagents)

중앙 오케스트레이터가 복잡한 태스크를 분해하고 전문 서브에이전트에게 위임한다.

```
[오케스트레이터]
    ├→ 서브에이전트 A (태스크 1)
    ├→ 서브에이전트 B (태스크 2)
    └→ 서브에이전트 C (태스크 3)
         ↓ (결과 수집)
    [오케스트레이터] → 통합 → 최종 출력
```

**의료 예시**:
```
오케스트레이터: "이 복잡한 환자 케이스 전체 평가"
→ 서브에이전트 1: "심장 관련 소견 정리"
→ 서브에이전트 2: "약물 최적화 검토"
→ 서브에이전트 3: "사회적 요인 평가"
→ 오케스트레이터: 세 결과를 통합해서 종합 계획 작성
```

**언제 쓰나**
- 태스크가 사전에 완전히 결정되지 않을 때
- 복잡도가 높아서 동적 분해가 필요할 때
- 다양한 전문성이 필요할 때

---

### 패턴 5: 평가-최적화 루프 (Evaluator-Optimizer)

AI가 결과를 생성하고, 다른 AI(또는 같은 AI)가 평가하고, 기준 미달이면 개선을 반복한다.

```
[생성 AI] → [평가 AI] → {
    기준 통과 → 출력
    기준 미달 → [피드백] → [생성 AI]  ← 루프
}
```

**의료 예시**:
```
AI1이 임상 노트 초안 작성
→ AI2(평가자): "투약 정보가 빠졌고, 서명란이 없음"
→ AI1: 피드백 반영해서 재작성
→ AI2: "통과"
→ 의사에게 전달
```

**언제 쓰나**
- 품질 기준이 명확할 때
- 첫 번째 시도가 항상 완벽하지 않을 때
- 자동 품질 관리가 필요할 때

**주의점**: 루프 탈출 조건을 반드시 설정한다. 평가자가 너무 엄격하면 영원히 루프에 갇힌다.

---

### 패턴 선택 가이드

| 상황 | 추천 패턴 |
|------|----------|
| 단계가 명확하고 선형 | 체인 |
| 독립 태스크, 속도 중요 | 병렬화 |
| 입력 유형이 다양 | 라우팅 |
| 복잡한 태스크, 전문성 필요 | 오케스트레이터-서브에이전트 |
| 품질 보증이 중요 | 평가-최적화 |
| 위 여러 가지 조합 | 하이브리드 |

실제 시스템은 이 패턴들을 조합해서 만든다. 예를 들어 "라우팅 → 각 경로에서 체인 → 마지막에 평가-최적화"처럼.

패턴을 알면 새로운 문제를 만났을 때 "이건 어떤 패턴 조합으로 풀 수 있지?"를 빠르게 생각할 수 있다.
