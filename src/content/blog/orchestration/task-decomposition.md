## 한줄 요약
태스크 분해는 "무엇을 할 것인가"를 "어떻게 할 것인가"로 변환하는 과정이다 — AI가 실행할 수 있는 크기로 잘라야 한다.

## 본문

### 태스크 분해가 왜 어려운가

"이 환자 종합 평가 보고서 작성해줘"라고 AI에게 시키면 어떻게 될까?

단일 AI에게 넘기면:
- 어디서부터 시작할지 막막하다
- 필요한 정보를 어디서 어떻게 가져올지 모른다
- 복잡도가 너무 높아서 중간에 길을 잃는다
- 결과가 일관성 없이 나온다

태스크를 제대로 분해하면:
- 각 단계가 명확하다
- 병렬 실행 가능한 것과 순차 실행해야 하는 것이 구분된다
- 어디서 실패했는지 정확히 알 수 있다
- 각 단계를 독립적으로 개선할 수 있다

### 좋은 태스크 분해의 기준

원자성 (Atomicity)
하나의 태스크는 하나의 명확한 출력을 가져야 한다. "여러 가지를 조사하고 분석해서 결정해줘"는 하나의 태스크가 아니다.

나쁜 분해:
```
태스크 1: 환자 데이터 수집하고 분석해서 치료 계획까지 세워줘
```

좋은 분해:
```
태스크 1: 환자 EMR에서 최근 6개월 데이터 수집 → 출력: 구조화된 환자 데이터
태스크 2: 수집된 데이터로 감별진단 → 출력: 진단 후보 목록
태스크 3: 진단 바탕으로 치료 옵션 검색 → 출력: 치료 옵션 3가지
태스크 4: 옵션 비교해서 최적안 선택 → 출력: 권고 치료 계획
```

의존성 명시
어떤 태스크가 어떤 태스크의 결과를 필요로 하는지 명확히 한다.

```
태스크 1 (독립)      태스크 2 (독립)
     ↓                    ↓
태스크 3 (태스크 1, 2 필요)
     ↓
태스크 4 (태스크 3 필요)
```

태스크 1과 2는 병렬 실행 가능. 태스크 3은 둘 다 끝나야 시작.

적정 크기
너무 크면 AI가 헤맨다. 너무 작으면 오버헤드가 커진다.

경험칙: 하나의 태스크는 하나의 LLM 호출로 완료할 수 있는 크기.

### 태스크 분해 기법

하향식 분해 (Top-down)
최종 목표에서 시작해서 점점 구체화한다.

```
목표: 당뇨 환자 연간 관리 보고서

1단계 분해:
- 임상 데이터 수집
- 치료 경과 분석
- 목표 달성도 평가
- 향후 계획 작성

2단계 분해 (임상 데이터 수집):
- HbA1c 추이 (최근 4회)
- 혈압/체중 추이
- 합병증 스크리닝 결과
- 약물 복용 이력
```

의존성 그래프 작성

```python
tasks = {
    "collect_emr": {"depends_on": []},
    "collect_labs": {"depends_on": []},
    "collect_meds": {"depends_on": []},
    "analyze_trends": {"depends_on": ["collect_emr", "collect_labs"]},
    "check_interactions": {"depends_on": ["collect_meds", "analyze_trends"]},
    "generate_report": {"depends_on": ["analyze_trends", "check_interactions"]}
}

# 병렬 실행 가능: collect_emr, collect_labs, collect_meds
# 순차 실행: analyze_trends → check_interactions → generate_report
```

### 동적 태스크 분해

처음부터 모든 태스크를 정해두지 않고, 실행하면서 다음 태스크를 결정하는 방식이다. ReAct (Reasoning + Acting) 패턴이 대표적이다.

```
목표: 이 환자에게 적합한 당뇨 치료제 찾기

[생각] 우선 현재 신기능을 확인해야 한다
[행동] get_lab_result("creatinine", "eGFR")
[관찰] eGFR = 42 mL/min/1.73m²

[생각] eGFR < 45이면 메트포르민 금기다. 다음 옵션 검색
[행동] search_guideline("T2DM treatment eGFR<45")
[관찰] SGLT2i도 eGFR < 45에서 혈당강하 효과 감소

[생각] GLP-1 RA가 적합할 수 있다. 심혈관 위험도 확인
[행동] calculate_risk("ASCVD", patient_data)
...
```

다음에 무슨 태스크를 할지를 이전 관찰 결과가 결정한다. 유연하지만 루프에 빠질 수 있어 최대 반복 횟수 제한이 필요하다.

### 의료 오케스트레이션에서의 핵심 원칙

사람 확인 지점 설정

모든 태스크를 AI가 자동으로 처리하지 않는다. 중요한 결정 전에 의사의 확인을 받는 지점을 명시적으로 설계한다.

```
[자동] 데이터 수집
[자동] 초기 분석
[사람 확인] "이 환자에게 침습적 시술을 권고할까요?" → 의사 승인
[자동] 승인 후 세부 계획 작성
[자동] 보고서 생성
```

감사 로그

어떤 에이전트가 어떤 순서로 어떤 판단을 했는지 전부 기록한다. 나중에 "왜 이런 결과가 나왔나"를 추적할 수 있어야 한다.
