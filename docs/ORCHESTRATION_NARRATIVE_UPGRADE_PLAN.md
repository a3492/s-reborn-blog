# 오케스트레이션 시리즈 — 서사·메시지 강화 기획서

> 목표: **백과사전형 정의 나열**에서 **한 줄기 스토리 안에서 결정이 내려지는 글**로 옮긴다.  
> 대상: `src/content/blog/orchestration/*.md` (시리즈 «RoundPrep»).

---

## 1. 진단 (왜 몰입·메시지가 약하게 느껴지는가)

| 현상 | 원인 | 독자에게 주는 인상 |
|------|------|-------------------|
| 스토리라인이 힘없다 | RoundPrep 인용이 **상단 블록 1~2문단**으로 끝나고, 본문은 **개념 설명만** 이어짐 | “예시와 본문이 다른 글 두 개가 붙어 있다” |
| 말하고자 하는 바가 흐리다 | 제목·`description`·`한줄 요약`이 **비슷한 말을 반복**하고, **판단 한 줄(테제)** 가 없음 | “읽고 나서 회의에서 뭐라고 말하면 되지?” |
| 너무 짧다 | 장(章)당 **장면·갈등·실패·재시도**가 없이 개념만 요약 | 노트 초안·슬라이드 불릿 수준 |
| 임팩트 부족 | **무엇이 걸려 있는지(스테이크)** 가 추상적(“품질”, “비용”) | 감정·시간·신뢰 같은 구체적 대가 없음 |

**한 줄 결론:** 지금 구조는 “용어집 + 도구 소개”에 가깝고, **에피소드가 주인공을 밀어 주지 않는다.**

## 2. 기획 방향 — “한 편 = 한 번의 회의/한 번의 장애/한 번의 승인”

### 2-1. 메시지를 임팩트 있게 만드는 공식 (편당 필수)

1. **테제(Thesis)** — 반박 가능한 한 문장을 **본문 첫 화면 안**에 둔다.  
   - 나쁜 예: “오케스트레이션을 정리한다.”  
   - 좋은 예: “상부에게 ‘AI 한 번이면 되죠?’라고 들었을 때, **노(No)** 를 **아키텍처 한 장**으로 말할 수 있어야 한다.”

2. **스테이크(Stake)** — 시간·돈·신뢰·환자 안전·책임 중 **하나만 골라 숫자나 상황**으로 박는다.  
   - 예: “회진 전 40분 → 알림만 정리해도 12분”처럼 범위를 좁힌다.

3. **전환점(Turn)** — 중간에 **한 번은 반드시 실패·오해·되돌림**이 나온다. (개념 글도 ‘잘못 설계했을 때’ 장면)

4. **지불(Payoff)** — 마지막에 RoundPrep 팀이 **다음 회의에서 할 말 한 줄**로 되먹인다.

5. **다음 화 훅** — 다음 `series_order` 글과 **갈등이 이어지는 한 문장**으로 끝낸다.

### 2-2. 글 유형별 권장 뼈대 (복붙용)

**A. 개념 허브·비교 (what-is, vs-agents, workflow-patterns 등)**  

1. RoundPrep **장면** 12~20문장 (누가, 어디서, 무슨 압박)  
2. **테제** 1문장  
3. 개념 본론 (기존 불릿은 유지 가능하되 각 절마다 “RoundPrep에 대입하면?” 1~2문장)  
4. **반례** 1개 (이 테제가 틀리는 조건)  
5. **페이오프** — 김서진/이혜준이 상부·정보보호·간호와 말할 **한 문장**  
6. 다음 글 훅

**B. 패턴·기술 (ReAct, 병렬, 상태, 컨텍스트 등)**  

1. “RoundPrep v0.x에서 **실제로 터진 일**” 8~15문장 (가상이어도 구체적으로)  
2. 테제  
3. 원리 설명  
4. “그래서 RoundPrep 파이프라인의 **어느 모듈**에 붙는가” 다이어그램 한 컷 분량 설명  
5. 체크리스트  
6. 페이오프 + 훅

**C. 도구 입문 (LangGraph, CrewAI, AutoGen)**  

1. 팀이 **왜 지금 이 도구를 열어봤는지** (대안 1개 거절 이유 포함)  
2. 테제: “이 도구가 해결하는 **한 가지 고통**”  
3. 의사 언어 비유 1개 + 최소 기능 지도  
4. RoundPrep에 **쓰면 좋은 지 / 과한 지**  
5. 페이오프 + 훅

**D. 운영·리스크 (보안, 테스트, 디버깅, 모니터링, 비용)**  

1. **사고 또는 거의 사고** 시나리오 (로그 없음, 비용 폭주, 프롬프트 인젝션 등)  
2. 테제  
3. 원칙 + 체크리스트  
4. “RoundPrep이 프로덕션에 가려면 여기서 막힌다” 한 단락  
5. 페이오프 + 훅

**E. 의료 케이스 스터디**  

- 이미 밀도가 있음 → **서사 프레이밍**만 강화: “누구의 승인이, 언제, 무엇을 막았는지” 한 타임라인 표 추가 권장.

## 3. 분량·리듬 가이드 (짧다는 느낌을 없애기)

| 구간 | 권장 분량 (한국어 본문 기준) | 역할 |
|------|---------------------------|------|
| 시리즈 단락 + 들어가며 + 테제까지 | 약 600~1,000자 | 몰입·스테이크 고정 |
| 본론 (개념/비유/도표) | 약 2,000~3,500자 | 신뢰·근거 |
| 반례 + 체크리스트 + 페이오프·훅 | 약 600~1,200자 | “그래서 뭐 하라고?” 해소 |

**합계 목표: 편당 약 3,500~5,500자** (유형 D·E는 상한 더 높여도 됨).  
`read_time`은 수정 후 실제 단어·문단 기준으로 다시 산정한다.

## 4. 시리즈 레일 고정 (RoundPrep 바이블 1페이지)

에디터·작성자 공용으로 **한 파일**에만 유지해도 된다 (예: `docs/ORCHESTRATION_ROUNDPREP_BIBLE.md` — 필요 시 후속 작업).

**최소로 적어 둘 것:**

- 등장인물 3~4명 (직책·성격·한 줄 목표)  
- 병동/권한/EMR 가정 (가상이면 “가정” 명시)  
- PoC 범위 (몇 병상, 어떤 데이터는 금지)  
- 분기 **타임라인** (1주차: 상부 질문 → 2주차: FHIR 막힘 → …)  

이걸 두면 매 글마다 세계관을 다시 쓰지 않아도 **한 줄기**가 유지된다.

## 5. `key_takeaways` 정책 변경

- 첫 번째 불릿: **description 복붙 금지** → **테제와 다른 말로 “행동/판단”** (한 문장).  
- 두 번째: 독자가 **내일 회의에서 인용할 수 있는 문장**.  
- 세 번째: `cu` 타임스탬프 줄 유지.

## 6. 실행 로드맵 (우선순위)

| 단계 | 범위 | 완료 기준 |
|------|------|-----------|
| P0 | RoundPrep 바이블 1p + 허브 3편 (order 1,2,16) | 테제·스테이크·페이오프·훅이 **모두** 있음 |
| P1 | order 3~8 (기초 패턴 축) | 각 편에 **실패 장면** 1개 이상 |
| P2 | order 9~15 (멀티에이전트·도구) | “왜 이 도구인가” **거절 서사** 포함 |
| P3 | order 17~21 (운영·비용) | **사고/거의 사고** 도입부 고정 |

진행 상황은 `docs/ORCHESTRATION_REVISION_LOG.md`에 날짜별로 남긴다.

## 7. 성공 정의 (기획 관점)

- 처음 읽는 사람이 **한 편만 읽고도** “RoundPrep이 뭔 일을 겪는 프로젝트인지” 말할 수 있다.  
- 임상의가 **개념 글을 읽고** 다음 날 IT와 **한 가지 요구**(로그, 승인 게이트, 재시도 상한 등)를 **문장으로** 꺼낼 수 있다.  
- 시리즈 순서대로 읽으면 **같은 갈등이 심화**된다 (매 편이 제로에서 시작하지 않는다).
