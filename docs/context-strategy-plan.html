Context Strategy

S-Reborn AI Blog 작업 컨텍스트 운영 계획

현재처럼 기능 축이 계속 늘어날 때는 “한 세션에 다 기억시키는 방식”보다, 작업 단위로 분리하고, 오래 유지되는 상태는 파일과 브랜치로 남기는 방식이 더 안정적입니다. 이 문서는 현재 Windows 워크스페이스 기준으로 실제로 쓸 수 있는 조합만 정리합니다.

기준: Windows + Astro + Supabase 목표: 컨텍스트 소진 방지 추천: worktree 우선

결론

지금 환경에서 가장 효율적인 조합은 이렇습니다.

  • Git worktree를 1순위로 사용합니다. 기능별 작업 공간을 물리적으로 분리하면 컨텍스트가 덜 섞이고, 되돌리기와 비교가 쉬워집니다.
  • tmux는 WSL, 리모트 Linux, macOS 같은 Unix 환경에서만 보조적으로 씁니다. Windows PowerShell 중심이라면 주력 도구로는 비효율적입니다.
  • cmux는 현재 Windows 주력 워크플로에 맞지 않습니다. 공식 저장소도 Windows는 아직 coming soon 상태라, 지금 당장 채택할 이유가 약합니다.
  • Google AI Studio는 로컬 편집기 대체가 아니라, 긴 문맥 분석과 설계 검토용으로 씁니다. 특히 대형 스냅샷, API 문서, 기획서를 한 번에 넣는 작업에 유리합니다.

도구 우선순위

실제 이 저장소에 맞춘 순서입니다.

1. Git worktree 브랜치별 작업 분리. 가장 안정적이고 즉시 적용 가능.
2. Google AI Studio 긴 문맥 검토, 파일 검색, 코드 실행, 설계 리뷰.
3. tmux WSL/리모트 환경에서만 세션 유지용으로 사용.
4. cmux Windows 지원이 아직 미완성이라 보류.

권장 운영 방식

컨텍스트가 거의 찼을 때는 세션을 길게 끌지 말고 상태를 파일에 남깁니다.

  • 작업 단위 분리: `public ui`, `admin ui`, `db schema`, `diff/detail pages`처럼 축을 쪼갭니다.
  • worktree 1개 = 축 1개: 홈 UI와 admin 상세 페이지를 같은 작업 공간에서 동시에 깊게 건드리지 않습니다.
  • handoff 파일 유지: `docs/`에 현재 상태, 미완 항목, 다음 축을 적어 둡니다.
  • 긴 분석은 AI Studio: 전체 설계, 대형 문서, API 문서 비교, RAG성 검색은 AI Studio가 잘 맞습니다.

권장 조합

현재 저장소 기준으로 현실적인 조합입니다.

main worktree
  ├─ public-site worktree
  ├─ admin-recovery worktree
  ├─ db-migration worktree
  └─ docs/planning worktree

long-context review
  └─ Google AI Studio (Gemini 3 Pro Preview / 2.5 Pro)

unix session persistence
  └─ tmux only when running in WSL or remote Linux

Google AI Studio를 쓰는 이유

Google AI Studio는 빠른 시작, 1M 토큰급 컨텍스트, File Search, Code Execution, Build mode를 제공하므로, 현재처럼 “문맥이 길고 파일이 많은 프로젝트”의 설계 검토에 적합합니다. 다만 로컬 편집기 대체가 아니라, 긴 문서와 상태를 한 번에 넣어서 설계 판단을 받는 역할이 더 맞습니다.

  • Long context: 1M+ 토큰 창을 활용해 기획서, 변경 로그, 스키마를 한 번에 검토합니다.
  • File Search: 문서와 스키마를 검색 가능한 지식베이스처럼 다룹니다.
  • Code Execution: 계산, 변환, 비교 결과를 스스로 검증하게 합니다.
  • Build mode: 빠른 프로토타입에는 좋지만, 현재 리포지토리처럼 계속 커지는 프로젝트의 주력 편집 환경으로 쓰기엔 제약이 있습니다.

권장 다음 액션

  • 현재 브랜치를 정리한 뒤 `public-site`, `admin-recovery`, `db-migration` 각각을 별도 worktree로 분리합니다.
  • 각 worktree마다 `docs/hand-off.md`를 두고 현재 상태와 다음 축을 기록합니다.
  • 긴 설계 검토가 필요할 때만 Google AI Studio에 요약본과 핵심 파일을 넣습니다.
  • tmux는 WSL/리모트 Linux로 넘어갈 때만 선택적으로 도입합니다.
