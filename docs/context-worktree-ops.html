Worktree 운영 규칙

컨텍스트가 찼을 때는 새 기능보다 분리부터 한다.

이 문서는 현재 S-Reborn AI Blog 작업에 맞춘 실제 운영 규칙이다. 핵심 원칙은 단순하다. 한 작업트리에 모든 변경을 몰아넣지 말고, 목적별 worktree를 분리해서 컨텍스트 소모와 롤백 비용을 줄인다.

Default rule
하나의 worktree = 하나의 책임.

공개 사이트, 관리자, DB 마이그레이션, 문서를 같은 트리에서 섞지 않는다.

1. 분리 기준

  • public-site: 공개 화면과 스타일만 다룬다.
  • admin-recovery: 관리자 UI, 권한, 발행, 로그, 복구 액션을 다룬다.
  • db-migration: Supabase schema, RLS, migration만 다룬다.
  • docs: 계획서, 인수인계, 작업 원칙만 다룬다.

2. 추천 명명 규칙

codex/public-site codex/admin-recovery codex/db-migration codex/docs-handbook

브랜치 prefix는 고정하고, worktree 이름은 역할이 바로 보이게 짧게 유지한다.

3. 시작 순서

1. 현재 상태를 고정한다.진행 중인 작업을 커밋하거나 임시 보관한다.
2. 새 worktree를 만든다.기능별로 다른 디렉터리에 체크아웃한다.
3. handoff 파일을 둔다.현재 목적, 파일 범위, 다음 액션을 적는다.
4. 세션을 분리한다.편집기, 터미널, AI 검토를 worktree 단위로만 운영한다.

4. Hand-off 템플릿

goal: settings/media diff 화면 완성 scope: src/pages/admin/settings/*, src/pages/admin/media/* done: 목록/상세/복구 구현 next: diff compact UI 정리 risks: audit snapshot schema 불일치 확인 owner: Codex

이 템플릿은 worktree마다 복붙해서 사용한다. 컨텍스트를 말이 아니라 파일로 남기는 방식이다.

5. 운영 원칙

상황 권장 동작 피해야 할 것
공개 UI 수정 public-site worktree만 사용 admin/db 작업과 같은 트리에 섞기
마이그레이션 추가 db-migration worktree에서만 처리 UI 수정과 같은 커밋에 넣기
긴 설계 검토 Google AI Studio에 요약본만 올리기 전체 저장소를 무작정 던지기
Unix/WSL 병렬 작업 tmux로 panes만 보조적으로 사용 Windows PowerShell에서 tmux를 주력처럼 쓰기

6. 병합 기준

  • 한 worktree는 한 목적만 끝내고 병합한다.
  • 병합 전에 build와 최소한의 화면 확인을 끝낸다.
  • 충돌이 생기면 오래된 worktree를 먼저 버리지 말고, 기준 worktree를 정한 뒤 다시 동기화한다.

7. 실무 결론

지금 이 프로젝트에서는 Git worktree가 핵심이다.
tmux는 보조, cmux는 현재 환경의 우선순위가 아니다.
Google AI Studio는 대용량 문맥 검토용으로만 쓰는 편이 맞다.
